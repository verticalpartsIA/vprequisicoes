## 1. Diagnóstico do Problema — Chunks 404

### Causa Raiz Confirmada
O Hostinger JS App executa `npm install` + build em diretório temporário, depois copia apenas os arquivos fonte para o diretório live. O `.next` do build temporário não vai para o live.

Com o guard `!.next/BUILD_ID` no `server.js`, o servidor encontrava o BUILD_ID antigo e **pulava o rebuild** → chunks com hashes velhos servidos → HTML referenciava hashes novos → 404.

### Fix Aplicado — `server.js`
Restaurado para o guard de BUILD_ID (fallback inteligente), mas agora o **"Comando de construção"** do Hostinger foi configurado para `npm run build`. Isso garante que o build roda no diretório live antes do `node server.js` iniciar.

### Fix Aplicado — `package.json` postinstall
Removido o build do postinstall (era redundante e rodava no diretório errado):
```json
"postinstall": "echo 'postinstall ok'"
```

### Fix Aplicado — `next()` constructor
Adicionado `dir: __dirname` para garantir que o Next.js encontra o `.next/` relativo ao `server.js`, independente do `process.cwd()` do Hostinger:
```js
const app = next({ dev: !isProd, hostname, port, dir: __dirname });
```

### Fix Aplicado — AuthGuard
Removido `loading` do deps array do `useEffect`. Causava re-execução infinita que disparava timeout de 10s:
```tsx
}, [pathname, router, searchParams]); // era: [..., loading]
```
