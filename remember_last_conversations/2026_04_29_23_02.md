## 2. Problema Principal — `__next_error__` / Chunks 404

### Causa Raiz
O Hostinger JS App faz **fresh clone + `npm install`** a cada deploy.
O `postinstall` roda `next build` → gera **novos hashes de chunks**.
O browser tinha o **webpack runtime antigo** (mapa de chunks) em cache.
O mapa antigo referenciava chunks com hashes velhos → arquivos não existem mais no servidor → **404**.

Hard refresh (Ctrl+Shift+R) funcionava porque baixava tudo do zero.

### Arquitetura do Deploy no Hostinger
- **Tipo:** JS App (Git-based, não FTP)
- **Entry:** `node server.js`
- **Build:** via `postinstall` no `package.json` (roda `next build` se `.next/BUILD_ID` não existe)
- **Sem `build_script` configurado** no Hostinger — a build acontece 100% via postinstall
- Cada deploy gera chunks com hashes diferentes → problema estrutural de cache

### Tentativas de Fix e Erros Cometidos

#### Commit `893bdc4` — Primeira tentativa (parcialmente correta)
Adicionou `headers()` no `next.config.js`:
```js
// /:path* → no-cache (HTML)
// /_next/static/:path* → immutable
```
**Problema:** `/:path*` e `/_next/static/:path*` ambos casavam, e o segundo não sobrescrevia o primeiro de forma garantida.

#### Commit `0838286` — Segunda tentativa (ERRO GRAVE)
Trocou para `/((?!_next).*)` (lookahead negativo).
**Problema crítico:** `path-to-regexp` (usado pelo Next.js) não suporta lookahead negativo → **servidor derrubado em runtime**. Site ficou completamente fora do ar por ~4 deploys.

#### Commit `662024e` — Fix correto
Voltou para padrão seguro e válido:
```js
[
  { source: '/:path*',             headers: [no-cache] },   // aplica em tudo
  { source: '/_next/static/:path*', headers: [immutable] }, // sobrescreve (vem depois)
]
```
Next.js aplica headers em ordem — o último match para o mesmo path ganha.
**Servidor voltou ao ar.**

### Estado Final do Cache Fix
- HTML (páginas): `no-cache, no-store, must-revalidate` — browser nunca cacheia
- `_next/static/**`: `public, max-age=31536000, immutable` — cacheado para sempre (hashes garantem unicidade)
- **Após um Ctrl+Shift+R do usuário**, o problema não recorre mais

## 3. Crash `TypeError: total_spent is undefined`

### Problema
`TopSuppliersTable.tsx` linha 44 chamava `.toLocaleString()` direto em `s.total_spent` sem null check.
Quando um fornecedor não tem cotações associadas, o campo vem `undefined` → crash no dashboard.

### Fix — Commit `6934bbd`
**Arquivo:** `src/components/dashboard/TopSuppliersTable.tsx`
```tsx
// Antes
{s.total_spent.toLocaleString('pt-BR', { minimumFractionDigits: 2 })}

// Depois
{(s.total_spent ?? 0).toLocaleString('pt-BR', { minimumFractionDigits: 2 })}
```

## 7. Lição Aprendida

**Nunca usar lookahead/lookbehind em `source` de `headers()` no `next.config.js`.**
O Next.js usa `path-to-regexp` que não suporta essa sintaxe. O padrão correto para sobrescrever headers em sub-paths é usar **ordem dos entries** (geral primeiro, específico depois).
